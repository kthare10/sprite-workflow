About detailed code-output relationship and potential migrating idea,

The `experiment1` orchestrator uses a state-driven pattern: stage execution is decided at runtime from database state plus filesystem markers (`_SUCCESS/_PARTIAL`).  
`gwas-qc-workflow` uses a static DAG pattern: task dependencies are declared upfront and then executed by the scheduler.  
The key distinction is whether runtime state checks are part of orchestration control flow.

2. Migration Design (toward a gwas-qc-workflow-style structure)

- Recommended organization  
  - Orchestration layer: add a `workflow_generator.py` that only defines DAG, dependencies, resources, retries  
  - Stage execution layer: add `bin/*.py` wrappers (one wrapper per stage, each calling existing modules)  
  - Business logic layer: keep existing implementations in `federated_learning` and `model_compare_tool`

- Stage decomposition idea: We can organize the overall workflow in `workflow_generator.py` and incorporate the implementation migrated from SPRITE to `bin/*.py`, as outlined below.
  - `plan_spans`  
  - `inventory`  
  - `freeze`  
  - `preproc`  
  - `snapshot`  
  - `enqueue_submit`  
  - `poll_retry`  
  - `model_compare_run`  
  - `visual_compare`  
  - `finalize_report`

- Key constraints  
  - Keep business truth in `S3_V2/_meta/sprite.sqlite` + manifests + `_SUCCESS/_PARTIAL`  
  - Separate scheduler state (DAG/task lifecycle) from business state (data readiness/quality gates)  
  - Keep window/span expansion configuration-driven, not hardcoded in DAG structure
